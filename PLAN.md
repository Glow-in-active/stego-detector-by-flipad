# План: детектор stego (mas_GRDH) через FLIPAD/DeepSAD

## 0) Цель и постановка

Мы хотим построить **детектор**, который по изображению решает:

- **inlier (stego)**: изображение получено нашим процессом `mas_GRDH` (Stable Diffusion v1.5 + stego-мэппинг/инициализация латента);
- **outlier (не-stego)**: изображение получено **не** этим процессом (датасет GenImage, желательно подмножество Stable Diffusion; в “open-world” допускаются и другие генераторы/источники).

Это соответствует постановке FLIPAD как **single-model attribution в open-world**, но в отчёте формулируем проще: **stego vs не-stego**.

Почему это может работать:

- `mas_GRDH` меняет распределение латентов/шумов (через stego mapping), и это может оставлять **слабый, но систематический** след в изображении;
- FLIPAD рассматривает задачу как **anomaly detection**: обучаемся на “норме” (наши stego), а всё остальное считаем “аномалией”.

Ключевое ограничение:

- у нас **нет пар** “тот же prompt/seed, но без stego”, поэтому модель может частично учиться отличать домены/пайплайны. Мы снижаем риск, выбирая “аномалии” из **похожего домена** (GenImage/StableDiffusion, 512×512, PNG).

## 1) Что именно в `flipad` мы используем

В репозитории `flipad/` есть готовый пайплайн `sma_deepsad.py`, который:

- извлекает признаки `feat` из изображений;
- обучает **DeepSAD** как модель anomaly detection;
- выбирает пороги по валидации при заданных **tolerable-fnr** (ложноотрицательная доля для inlier);
- тестирует на разных “других” директориях.

Важно: в исходном FLIPAD “inlier” обычно соответствует “сгенерировано целевым генератором”, а “outlier” — “не им”.
В нашем проекте:

- **inlier** = `mas_GRDH stego`
- **outlier** = `genimage`

### 1.1) Как устроены данные у `flipad`

`flipad/sma_deepsad.py` ожидает папки вида:

```
data/<dataset>/<train_dir>/{train,val,test}/*.png
data/<dataset>/<real_dir>/{train,val,test}/*.png
```

В оригинале `real_dir` — это “реальные” изображения. Для нас это просто “не-stego”.

Мы заведём `dataset` как **одну строку** (удобно переиспользовать существующий выбор `coco2014train`),
а имена директорий:

- `<train_dir>` = `mas_grdh_stego`
- `<real_dir>` = `genimage`

Тогда команды `sma_deepsad.py` можно использовать почти без изменений.

## 3) Метрики и что оптимизируем

Приоритет: **не пропустить stego**.

Это значит:

- выбираем целевой **TPR** (например, 99% или 95%) на inlier-классе (stego);
- под этот TPR выбираем порог по валидации;
- на test-сете считаем, сколько outlier (genimage) ошибочно прошло как inlier:
  - **FPR при TPR=99%** (главная метрика)

Дополнительно (для отчёта удобно):

- **AUROC** (общая разделимость по всем порогам)
- confusion matrix при выбранном пороге

Терминология:

- TPR = 1 − FNR
- FNR задаётся в `--tolerable-fnr`
  - пример: `FNR=0.01` соответствует `TPR=99%`

## 4) Экспериментальный план (по шагам)

### Шаг A — Сборка датасета на диске (единая структура)

Сформировать **локальную структуру** (Colab или scratch на кластере), которую ожидает `flipad`:

```
data/coco2014train/
  mas_grdh_stego/
    train/  (N_train)
    val/    (N_val)
    test/   (N_test)
  genimage/
    train/  (N_train)  # опционально, нужно для текущего кода sma_deepsad.py
    val/    (N_val)    # опционально
    test/   (N_test)
```

Почему “genimage/train и val” всё равно нужны:

- `sma_deepsad.py` в обучении формирует `X_train` как конкатенацию признаков для:
  - `train_dir/train` (inlier)
  - `real_dir/train` (outlier)
- поэтому для запуска “как есть” нужно заполнить оба.

Если мы хотим строгий “one-class” без outlier в обучении — это отдельная модификация кода.
На первом этапе используем “как есть”, чтобы быстрее получить работающий результат.

Рекомендуемое разбиение (с учётом текущих объёмов stego≈214k, genimage=166k):

**Для полного прогона (суперкомп):**

- **train**: 130k stego + 130k genimage (баланс)
- **val**: 10k stego + 10k genimage (для порога и совместимости с текущим `sma_deepsad.py`)
- **test**: 20k stego + 26k genimage (остаток) *или* наоборот:
  - **test**: 20k stego + **все оставшиеся genimage** (до 166k), если хотим максимально точный FPR.

**Для пилота (Colab):**

- stego: 5k train / 2k val / 5k test
- genimage: 5k train / 2k val / 20k test (или 5k test, если сеть/диск ограничены)

### Шаг B — Быстрый пилот (проверка, что сигнал вообще есть)

До того, как гонять 200k/200k, делаем пилот:

- stego: 5k train, 2k val, 5k test
- genimage: 5k train, 2k val, 20k test

Смысл:

- быстро понять, разумна ли FPR при TPR=99%;
- подобрать `feat` и базовые гиперпараметры.

### Шаг C — Выбор признаков (`feat`) и гиперпараметров

В `sma_deepsad.py` есть варианты:

- `--feat raw`: базовые признаки по пикселям (с resize, для SD ставит `resize_to=128`)
- `--feat dct`: DCT-признаки (аналогично)
- `--feat act`: “FLIPAD-ветка” с реконструкцией активаций и downsampling

Для проекта FLIPAD-логичнее стартовать с:

- `--feat act`
- `--downsampling center_crop` (как в их `sd.sh`)
- `--net-name cifar10_biglenet` или `lenet_128x128` (надо подобрать по памяти/скорости)
- `--num_select_channels 3` (для пилота) → потом 32/64/128

Почему начинать с малого количества каналов:

- признаки `act` для SD имеют 128 каналов и пространственную размерность, их обработка тяжелее;
- на пилоте важнее скорость итераций.

### Шаг D — Калибровка порога под “не пропустить stego”

Мы хотим TPR=99% по stego.
В `sma_deepsad.py` это задаётся списком `--tolerable-fnr`.

Например:

- `--tolerable-fnr 0.01`  → порог под TPR≈99%

В отчёте фиксируем:

- какой порог получился по `val` на stego,
- какая при этом FPR на `genimage/test`.

### Шаг E — Масштабирование на 200k/200k

Когда пилот показал, что FPR приемлема:

- увеличиваем `--num-train`, `--num-val`, `--num-test`
- включаем кеширование результатов по умолчанию (не ставим `--avoid-caching`)
- аккуратно выбираем `batch-size` и `num-workers` под файловую систему кластера

## 5) Параллелизация на 4 GPU (V100/A100)

### 5.1) Где реально узкое место

В пайплайне есть два тяжёлых этапа:

1) **вычисление признаков** для большого числа изображений (feature extraction);
2) **обучение DeepSAD** (обычно легче).

Чаще всего выгоднее распараллелить именно (1).

### 5.2) Практичный способ “в репликах” (без DDP)

Делаем шардинг датасета и запускаем 4 независимых job’а:

- GPU0: обрабатывает shard_0
- GPU1: shard_1
- GPU2: shard_2
- GPU3: shard_3

Каждая job:

- считает признаки/кеш для своей части;
- сохраняет промежуточные файлы в отдельную папку кэша (например `output/cache_shard=k/`).

Потом:

- объединяем кэш (или просто запускаем финальный тренировочный прогон на 1 GPU уже с готовыми сохранениями).

Почему так:

- меньше инженерных рисков, чем DDP;
- хорошо ложится на SLURM/очередь;
- масштабируется линейно.

### 5.3) Если всё-таки DDP

Возможен вариант обучения DeepSAD через DDP, но это имеет смысл только если:

- признаки уже посчитаны,
- и обучение реально стало bottleneck’ом.

Скорее всего, в этом проекте DDP не даст большого выигрыша, поэтому оставляем как “опцию”.

## 6) Организация работы с S3 (когда дойдём до этого)

Данные лежат в S3, и **полностью выгрузить на локальный компьютер нельзя**. При этом текущие скрипты `flipad`
ожидают обычные директории на файловой системе (они делают `Path(...).iterdir()` и берут `*.png`).

Поэтому есть 3 рабочих сценария (по возрастанию сложности):

- **A (для пилота в Colab — рекомендуется)**: скачиваем **малый поднабор** из S3 в локальную ФС Colab
  в структуру `data/coco2014train/...` и прогоняем `flipad` без правок.
- **B (для суперкомпа, минимум правок)**: монтируем S3 бакет как файловую систему (например, `s3fs-fuse`/`goofys`)
  и снова используем `flipad` без правок, просто указывая `--data-path` на смонтированный путь.
- **C (самый гибкий, но требует кода)**: меняем `flipad` датасет/лоадер, чтобы он принимал список S3 ключей и
  читал изображения через `boto3`/`s3fs` напрямую (стриминг + локальный кэш).

Практическая рекомендация:

- **Colab**: сценарий A (скачать 2–10k+2–10k) — быстро, воспроизводимо, дешево по времени.
- **Суперкомп**: сценарий B (mount) если админы разрешают FUSE и сеть к S3 нормальная; иначе сценарий C.

### 6.1) Как организовать выборку из S3 (общая идея)

Ключевое правило: **не переносим и не “перемешиваем” файлы в S3** (это долго и не нужно).
Все “смешивание” делаем на уровне **списков ключей** (manifest), а на диск скачиваем только нужные подмножества.

Сырые данные в S3 (как есть):

- `s3://stegopractice/Genimage/ai/*.png` — **только Stable Diffusion** (по договорённости), возможны разрывы в нумерации.
- `s3://stegopractice/Stego/<папка_источника>/identity/*.png` — stego из `mas_GRDH`, рядом могут лежать `*.json` метаданные.
  - `*.json` **игнорируем**: текущие датасеты `flipad` читают только файлы с расширениями картинок.

Чтобы воспроизводимо формировать `train/val/test`, делаем:

- фиксированный `seed`;
- список ключей/путей в S3 для каждого класса;
- затем детерминированно режем на `train/val/test`.

Важно: для пилота достаточно скачать **только `train/val/test` подмножества**, а не весь бакет.

### 6.2) Сплиты без зависимости от нумерации (hash-split по S3 key)

Проблема “в каждой папке нумерация 0..N” и “в GenImage счётчик обрывается” не мешает, если мы **не используем номер**.
Мы делаем сплит по **полному пути ключа** (он уникален).

Схема:

- берём строку `key` (например `Stego/chatgpt/identity/00001234.png`);
- считаем детерминированный хэш от `(seed + key)`;
- переводим в диапазон 0..99:
  - 0–79 → `train`
  - 80–89 → `val`
  - 90–99 → `test`

Плюсы:

- не требуется копирование/переименование в S3;
- сплиты стабильные и воспроизводимые;
- исключаем утечки между сплитами по одному и тому же файлу.

### 6.3) Стратификация stego по подпапкам `Stego/`

В `Stego/` много подпапок (разные наборы промптов: `chatgpt/coco/flickr/laion/own…`).
Если просто “свалить всё в один список”, можно получить перекос доменов.

Поэтому делаем так:

- собираем ключи `*.png` **отдельно для каждой подпапки** (например, по префиксу `Stego/chatgpt/` и т.д.);
- внутри каждой подпапки применяем hash-split из 6.2;
- итоговые `train/val/test` — конкатенация сплитов по всем подпапкам.

Это гарантирует, что в каждом сплите присутствуют все источники промптов.

### 6.4) Безопасность ключей

Никаких ключей в репозитории/ноутбуках. В Colab:

- **Colab Secrets** или переменные окружения;
- для Yandex Object Storage обычно нужен `endpoint_url` и `addressing_style=path`.
